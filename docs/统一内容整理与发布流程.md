# 统一内容整理与发布流程

本项目统一处理“分享”和“讲道”两类资料。Codex 的日常内容入口使用“整理并发布分享或讲道资料”对话，正式工作目录必须是本机 SSD 上的 Git 仓库。

## 初次准备

```shell
python -m pip install -r requirements.txt
npm install
```

Windows 和 macOS 都使用 `python`、`npm` 命令，不使用 `npm.cmd` 或硬编码的 Git 路径。

收件目录和归档目录通过环境变量提供：

- `RONNIE_SHARE_INBOX`：分享收件目录
- `RONNIE_SERMON_INBOX`：讲道收件目录
- `RONNIE_SERMON_ARCHIVE`：教会讲道受保护归档目录

## 跨项目文档同步规则

讲道整理项目与个人网页项目共同维护讲道发布流程。以后只要新增或修改讲道整理、讲道翻译、讲道发布、NAS 归档、日期、元数据、发布前 MD、导出或质量门禁相关规则，都要同步检查两个项目的对应文档。若只更新其中一个项目，最终汇报必须说明另一个项目尚未同步及原因。

## 翻译原则

凡遇到需要翻译的资料，尤其是讲道 PDF、讲章、双语稿、英文资料或其他外文原文，必须以原文为准逐句完整翻译，不得用概括、摘要、意译式改写来代替翻译。

执行时必须保留原文的完整信息量，包括论点、例子、经文、引用、过渡句、重复强调、语气和结论。可以为了中文通顺调整语序，但不得删减内容、合并段落造成信息丢失，也不得自行补充原文没有的意思。

如果提取稿含多语言版本，应优先确认用户指定的翻译源；未指定时，先说明将依据哪一种语言版本翻译。机器提取稿只能作为底稿，发布前必须逐段核对，确保中文稿忠实覆盖原文。

新增翻译硬性规则：

- 原文已经写明引用经文出处时，中文翻译必须自然保留并译出对应经文章节，不得省略；例如原文写 Galatians 2:14，中文应保留为“加拉太书 2:14”。
- 中文正文中直接引用圣经经文时，默认全部使用《和合本》译文；如需解释英文词语、原文词义或其他译本差异，可在和合本经文之外另加说明，但不得用其他中文译本替代正文经文。

## 正文与日期质量门

生成 `data/processed/` 和 `src/content/posts/` 中的 Markdown 前，必须确认正文格式适合 Astro/Markdown 前台渲染：

- 段落之间必须有空行；单换行不会在公开页面形成独立段落。
- Admin 编辑区能看到换行，不代表前台会分段；以前台 Markdown 渲染结果为准。
- `[WD]`、`[SLIDE]`、`[slide]`、`[SLIDE - ...]`、`[TIMELINE SLIDE - ...]`、`[MAP]`、`[PAGE ...]` 等讲稿/投影片标记不得进入中文原稿、processed 或正式 posts。
- 若标记独占一行，删除整行；若标记后有正文，只删除标记，保留正文。
- 讲道内容的 `date` 一律使用本次整理发布时的日期；不得从原始文件名、讲道内容、历史讲道日期、文件修改时间或用户旧日期线索推断。补录旧讲道也按整理发布当天进入网站时间线。非讲道文章如用户明确指定发布日期，仍按对应内容流程处理。
- 从 `讲道整理` 项目导入的讲道，必须完成 raw/source 交接、processed 副本、posts 文件、NAS 归档状态记录；不能只复制 posts 文件后跳过 `archive-sermon`。NAS 归档内容只包含原始文件、英文原稿文件、翻译后的中文原稿文件，不归档发布前 MD、审计中间目录、网站 posts 或 processed 副本。

## 分享流程

0. 发布前同步远端：

   ```shell
   npm run sync
   ```

   这一步必须在任何文章修改、导出、构建、提交或推送之前执行。`npm run sync` 内部使用 `git pull --rebase origin main`，用于先把本地 `main` rebase 到远端最新状态，避免覆盖用户通过 Admin 后台、GitHub 网页或其他账号已经发布的内容。若本地已有未提交改动，必须先暂停并确认，不得跳过同步后继续发布。

1. 只读检查：

   ```shell
   python scripts/content_workflow.py inspect share
   ```

2. 将文件校验后移入 `data/raw/分享/`：

   ```shell
   python scripts/content_workflow.py ingest share
   ```

3. 预览整理正文并生成文章：

   ```shell
   python scripts/content_workflow.py publish share --source-file "data/raw/分享/<file>" --dry-run --description "人工概括型摘要。"
   ```

4. 确认标题、经文、摘要和目标路径无误后正式发布：

   ```shell
   python scripts/content_workflow.py publish share --source-file "data/raw/分享/<file>" --description "人工概括型摘要。"
   ```

5. 人工检查本次文章、整理报告和 Git 差异，然后执行构建、提交、推送和线上验证。

成功标准：收件目录不再保留已处理文件；原始文件 SHA-256 校验通过；只更新分享文章；线上页面包含正确标题、分类和正文。

## 讲道流程

0. 发布前同步远端：

   ```shell
   npm run sync
   ```

   这一步必须在任何讲道资料入库、文章生成、构建、提交或推送之前执行。`npm run sync` 内部使用 `git pull --rebase origin main`，用于先把本地 `main` rebase 到远端最新状态。若同步失败、发生 rebase 冲突，或本地已有未提交改动，必须暂停并与用户确认，不得继续发布。

1. 只读检查：

   ```shell
   python scripts/content_workflow.py inspect sermon
   ```

2. 确认日期、经文标题和讲员后入库：

   ```shell
   python scripts/content_workflow.py ingest sermon --date YYYYMMDD --title "经文与标题" --speaker "讲员"
   ```

3. 脚本把 PDF 移入 `data/raw/教会讲道/<日期标题_讲员>/`，进行 SHA-256 校验，并生成 `*.extracted.txt`。
4. Codex 检查提取质量，过滤页眉、页脚和异常字符，按“翻译原则”逐句完整翻译并校订为中文 TXT。机器提取稿不能直接发布，也不能把逐句翻译替换成概括式整理稿。
5. 将完成的整个讲道目录复制到受保护归档区：

   ```shell
   python scripts/content_workflow.py archive-sermon --folder "data/raw/教会讲道/<日期标题_讲员>"
   ```

6. 预览生成讲道文章和目录：

   ```shell
   python scripts/content_workflow.py publish sermon --folder "data/raw/教会讲道/<日期标题_讲员>" --dry-run --description "人工概括型摘要。"
   ```

7. 确认标题、讲员、经文、摘要、slug 和目标路径无误后正式发布：

   ```shell
   python scripts/content_workflow.py publish sermon --folder "data/raw/教会讲道/<日期标题_讲员>" --description "人工概括型摘要。"
   ```

   如需更新同一 source folder 对应的既有文章，必须显式加 `--update-existing`，且只能更新 registry 中记录的原 slug/path。

8. 检查本次文章、讲员、经文、分类和 Git 差异，然后执行构建、提交、推送和线上验证。

成功标准：原始 PDF 和中文 TXT 均已保存；项目与归档副本的 SHA-256、文件数和字节数一致；只更新讲道文章；线上页面包含正确标题、讲员、分类和正文。

## 安全边界

- `data/raw/` 只新增，不直接改写原始文件。
- 受保护归档区只新增，不移动、不删除、不覆盖。
- 任何目标已存在时，统一入口脚本都会停止，由用户或 Codex先核对，不自动覆盖。
- 分享发布必须指定 `--source-file`，讲道发布必须指定 `--folder`；不得无参数全量扫描 raw 目录。
- 正式发布前必须先运行 `--dry-run`，确认只会生成本次目标文章和报告。
- `description` 必须是人工概括型摘要，不允许正文截取、模板句或占位符。
- 讲道导入会维护 `docs/内容整理报告/sermon-import-registry.csv`，用于锁定 source folder、source SHA-256、slug、processed path 和 post path。
- 讲道经文若在 folder、文件名、正文前部之间冲突，脚本会停止，必须人工确认后再发布。
- 导入脚本不会清空 `src/content/posts/`，也不会删除另一类别的文章。
- 发布前必须先执行 `npm run sync`，确认本地 `main` 已通过 `git pull --rebase origin main` 同步到远端最新状态；没有完成同步，不得开始修改、导出、构建、提交或推送。
- 发布前必须检查 `git diff`，发布后必须验证线上页面。
- 发布前必须抽查正文段落空行和讲稿/投影片标记清理结果，尤其是讲道稿中的 `[WD]`、`[SLIDE]` 等来源标记。
- 讲道导入/导出脚本必须把 `date` 设为本次整理发布日；不得为了旧讲道而使用历史日期。非讲道文章的指定发布日期规则按对应内容流程处理。
- 讲道发布必须完成 NAS 归档或记录 `archive_status=pending/skipped_by_user`；未归档时不得在交接中写成完整完成。
